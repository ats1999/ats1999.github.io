<h2>What is WAL?</h2>
<p>WAL is a database mechanism that logs every changes made to database before applying them to database. It ensures durability and crash recovery.</p>
<h4>Key Points:</h4>
<ul>
<li>WAL logs are not human friendly</li>
<li>Used internally by databases (e.g., PostgreSQL, MySQL).</li>
<li>Focused on write performance, data integrity and recovery.</li>
<li>Logs every write (insert/update/delete) operation sequentially.</li>
<li>Typically not designed for external consumption.</li>
</ul>
<p>Use Case:
Database recovery, replication, and ensuring ACID compliance in transactional systems.</p>
<h2>What is CDC?</h2>
<p>CDC is a technique to capture changes made into the database and make them available to external systems in near real-time.</p>
<h4>Key Points:</h4>
<ul>
<li>CDC can be human friendly</li>
<li>CDC mostly used by external system to react on database changes(e.g notification, audit logs, analytics, etc)</li>
<li>Mostly CDC is used via WAL logs, but can be implemented without WAL using DB pooling, triggers, timestamps, etc</li>
</ul>
<hr>
<hr>
<p>Now we have understanding of WAL and CDC. Let's deep dive into WAL first, then we'll use WAL to setup CDC.</p>
<h2>Why DBMS uses WAL?</h2>
<p>The main reason for the databases to use WAL is performance and data integrity. To maintain data integrity, database systems needs to synchronously write (flush &#x26; sync) the updated page from shared memory to disk whenever a tuple is inserted, updated or deleted. However, this approach results in poor performance.</p>
<p>Database systems logs all changes and actions into WAL logs files. While actual tuple updates are kept in memory and synced with disk asynchronously. Database can use WAL logs to recover from crash, in case database crashed after writing to WAL logs but before synchronizing changes with disk.</p>
<p>WAL logs are also used for <a href="https://www.interdb.jp/pg/pgsql10.html">Online Backup and Point-In-Time Recovery (PITR)</a>, <a href="https://www.interdb.jp/pg/pgsql11.html">Streaming Replication</a>, <a href="https://www.postgresql.org/docs/current/two-phase.html">2PC</a>, etc.</p>
<h2>CDC, WAL &#x26; PostgreSQL</h2>
<p>The intent of CDC is to capture the changes made to the database, no matter how you achieve it.</p>
<p>There are different ways to implement CDC:</p>
<ul>
<li>Using WAL</li>
<li><a href="/blog/audit-logs/">Using timestamp column in the table and periodically polling the database</a></li>
<li><a href="/blog/audit-logs/">Using database trigger to write every update into another table</a></li>
</ul>
<p>In context of WAL, CDC is a human friendly decoding of WAL logs.</p>
<p>Let's understand few terms before we jump on setting up CDC</p>
<ul>
<li><a href="https://www.postgresql.org/docs/current/logicaldecoding-explanation.html">Logical Decoding</a> - It is a feature in PostgreSQL, that can be used to parse WAL logs into easy to use format.</li>
<li><a href="https://www.postgresql.org/docs/current/logicaldecoding-explanation.html#LOGICALDECODING-REPLICATION-SLOTS">Replication Slot</a> - Replication slot is a kind of queue which is used by PostgreSQL to stream changes to client. PostgreSQL replication slots should be unique to the client. We can create multiple replication slots if multiple clients required.</li>
<li><a href="https://www.postgresql.org/docs/current/logicaldecoding-explanation.html#LOGICALDECODING-EXPLANATION-OUTPUT-PLUGINS">Output Plugins</a> - Output plugins are used in conjunction with logical decoding to customize the format or mechanism of streaming the changes. They convert changes in WAL into a format that can be consumed by external systems.</li>
<li><a href="https://www.postgresql.org/docs/current/logical-replication-publication.html">Publication</a> - Publication defines which tables/changes are available to replicate</li>
</ul>

<p>You can see WAL logs contains LSN(log sequence number) and XID(transaction id).</p>
<ul>
<li>LSN is a unique position marker in the WAL stream. Every change recorded in WAL gets a new LSN. It tells you where exactly an operation is recorded. Think of it like a byte offset or a "timestamp" for the WAL logs. It’s important for replication, crash recovery, streaming, etc.</li>
<li>XID is an identifier for a transaction. Every transaction gets a unique XID. All changes from the same transaction share the same XID.</li>
</ul>
<h2>Recommendation</h2>
<h3>Debezium</h3>
<p>While <code>wal2json</code> seems simple to use, but i would recommend to use <a href="https://debezium.io/documentation/reference/3.1/architecture.html">debezium</a> in production. As it handle many edge cases which may occur in production if something goes wrong.</p>
<h3>TRUNCATE is not DELETE</h3>
<p>TRUNCATE quickly removes all rows from a set of tables, behave like DELETE. But, TRUNCATE does not emit <code>DELETE</code> operation in WAL for each row, instead it writes a single WAL log per TRUNCATE.</p>
<blockquote>
<p><code>wal2json</code> does not capture table level changes, so you will not see any data in WAL output for TRUNCATE when using <code>wal2json</code>. <code>pgoutput</code> plugin can be used to capture TRUNCATE. <code>pgoutput</code> can also be used with <a href="https://debezium.io/documentation/reference/3.1/architecture.html">debezium</a> if you are using debezium.</p>
</blockquote>
<h3>Table without primary key</h3>
<p><code>wal2json</code> or <code>debezium</code> generate event for only <code>INSERT</code>, when using a table without primary key.</p>
<p>You need to use replica identity feature to capture <code>UPDATE</code> or <code>DELETED</code></p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="sql" data-theme="github-dark-dimmed"><code data-language="sql" data-theme="github-dark-dimmed" style="display: grid;"><span data-line=""><span style="color:#F47067">ALTER</span><span style="color:#F47067"> TABLE</span><span style="color:#F47067"> &#x3C;</span><span style="color:#ADBAC7">table_name</span><span style="color:#F47067">></span><span style="color:#F47067"> REPLICA</span><span style="color:#F47067"> IDENTITY</span><span style="color:#ADBAC7"> FULL;</span></span></code></pre></figure>
<h3>Updating primary key</h3>
<p>PostgreSQL will execute first delete operation and then insert operation, when you update primary key of any row. You may need to handle this case in your application.</p>
<p>Let's say you are using email as primary key for users table. You have a logic in WAL consumer to delete data related to user when receiving delete record. So, if you don't consider this case, you'll end up deleting all users data even for email update.</p>
<h3><a href="https://www.postgresql.org/docs/current/storage-toast.html">TOAST</a> values</h3>
<blockquote>
<p>PostgreSQL uses a fixed page size (commonly 8 kB), and does not allow tuples to span multiple pages. Therefore, it is not possible to store very large field values directly. To overcome this limitation, large field values are compressed and/or broken up into multiple physical rows. This happens transparently to the user, with only small impact on most of the backend code. The technique is affectionately known as TOAST (or “the best thing since sliced bread”). The TOAST infrastructure is also used to improve handling of large data values in-memory.</p>
</blockquote>
<p>Since, TOAST values are not stored in WAL logs, you'll not receive them in WAL logs unless there is any update to that column. You may need to consider this case when using CDC.</p>
<h3>Hight Traffic &#x26; Low Traffic</h3>
<p>Consider a situation, where PostgreSQL instance contains multiple databases and one of them is a high-traffic database. Debezium captures changes in another database that is low-traffic in comparison to the other database. Debezium then cannot confirm the LSN as replication slots work per-database and Debezium is not invoked. As WAL is shared by all databases, the amount used tends to grow until an event is emitted by the database for which Debezium is capturing changes.</p>
<p>Solution -> <a href="https://debezium.io/documentation/reference/3.0/connectors/postgresql.html#postgresql-wal-disk-space">postgresql-wal-disk-space</a></p>
